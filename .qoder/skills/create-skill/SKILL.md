# 为 Qoder 创建技能

本技能引导你创建高效的 Agent 技能。技能是 markdown 文件，教会 agent 如何执行特定任务：按团队规范审查 PR、生成提交信息、查询数据库结构，或任何专业工作流。

## 开始之前：收集需求

在创建技能前，先收集以下关键信息：

1. **目标与范围**：技能要帮助完成什么具体任务或工作流？
2. **存储位置**：个人技能（`~/.qoder/skills/`）还是项目技能（`.qoder/skills/`）？
3. **触发场景**：agent 在什么情况下应自动应用该技能？
4. **领域知识**：agent 需要哪些它本身不具备的专业信息？
5. **输出格式**：是否有特定的模板、格式或风格要求？
6. **现有模式**：是否有现成的示例或规范可以参考？

### 从上下文推断

如果对话中有上下文，可从讨论内容推断技能。可基于对话中出现的工作流、模式或领域知识创建技能。

### 收集更多信息

如需澄清，可直接询问用户：
- "这个技能应该存放在哪里？" → 个人（`~/.qoder/skills/`）还是项目（`.qoder/skills/`）
- "技能是否需要包含可执行脚本？" → 是 / 否

## 技能文件结构

### 目录布局

技能以包含 `SKILL.md` 文件的目录形式存储：

```
skill-name/
├── SKILL.md              # 必需 - 主要指令文件
├── reference.md          # 可选 - 详细参考文档
├── examples.md           # 可选 - 使用示例
└── scripts/              # 可选 - 工具脚本
    ├── validate.py
    └── helper.sh
```

### 存储位置

| 类型 | 路径 | 作用范围 |
|------|------|---------|
| 个人 | `~/.qoder/skills/skill-name/` | 跨所有项目可用 |
| 项目 | `.qoder/skills/skill-name/` | 与仓库使用者共享 |

### SKILL.md 结构

每个技能必须包含 YAML 前置元数据和 markdown 正文：

```markdown
---
name: your-skill-name
description: 简要描述技能的作用和使用时机
---

# 技能名称

## 使用说明
清晰的分步骤指导。

## 示例
具体的使用示例。
```

### 必填元数据字段

| 字段 | 要求 | 用途 |
|------|------|------|
| `name` | 最多 64 字符，仅限小写字母、数字、连字符 | 技能唯一标识 |
| `description` | 最多 1024 字符，不可为空 | 帮助 agent 判断何时应用该技能 |

## 编写高质量描述

描述对技能发现**至关重要**，agent 依靠它判断何时应用技能。

### 描述最佳实践

1. **使用第三人称**（描述会被注入系统提示词）：
   - 好：处理 Excel 文件并生成报表
   - 避免：我可以帮你处理 Excel 文件

2. **具体并包含触发词**：
   - 好：从 PDF 文件提取文本和表格，填写表单，合并文档。在处理 PDF 或用户提到 PDF、表单、文档提取时使用。
   - 差：帮助处理文档

3. **同时包含"做什么"和"何时用"**：
   - 做什么：技能的具体能力
   - 何时用：agent 应使用该技能的触发场景

### 描述示例

```yaml
# PDF 处理
description: 从 PDF 文件提取文本和表格，填写表单，合并文档。在处理 PDF 或用户提到 PDF、表单、文档提取时使用。

# Git 提交助手
description: 通过分析 git diff 生成描述性提交信息。在用户请求帮助撰写提交信息或审查暂存更改时使用。

# 代码审查
description: 按照团队标准审查代码质量、安全性和可维护性。在审查 PR、检查代码变更或用户请求代码审查时使用。
```

## 核心编写原则

### 1. 简洁是关键

上下文窗口由对话历史、其他技能和请求共享，每个 token 都有竞争。

**默认假设**：agent 本身很聪明，只需补充它不具备的上下文。

反复审视：
- "agent 真的需要这个解释吗？"
- "agent 是否已经知道这些？"
- "这段内容值得占用 token 吗？"

### 2. SKILL.md 控制在 500 行以内

为获得最佳性能，主文件应保持简洁，详细内容使用渐进披露方式。

### 3. 渐进披露

将核心信息放在 SKILL.md，详细参考资料放在单独文件中，agent 按需读取：

```markdown
## 更多资源
- 完整 API 详情见 [reference.md](reference.md)
- 使用示例见 [examples.md](examples.md)
```

**保持一级引用深度**：从 SKILL.md 直接链接到参考文件，深层嵌套可能导致读取不完整。

### 4. 设置合适的自由度

| 自由度 | 适用场景 | 示例 |
|--------|---------|------|
| **高**（文字说明） | 多种有效方案，依赖上下文 | 代码审查指南 |
| **中**（伪代码/模板） | 有首选模式但允许变通 | 报表生成 |
| **低**（具体脚本） | 脆弱操作，一致性至关重要 | 数据库迁移 |

## 常用模式

### 模板模式

提供输出格式模板：

```markdown
## 报告结构

使用以下模板：

# [分析标题]

## 摘要
[一段概述关键发现]

## 关键发现
- 发现 1（含支撑数据）
- 发现 2（含支撑数据）

## 建议
1. 具体可操作建议
2. 具体可操作建议
```

### 示例模式

当输出质量依赖参考示例时使用：

```markdown
## 提交信息格式

**示例 1：**
输入：添加基于 JWT 的用户认证
输出：
feat(auth): 实现基于 JWT 的认证

添加登录端点和 token 验证中间件
```

### 工作流模式

将复杂流程拆解为清晰的步骤清单：

```markdown
## 表单填写工作流

复制此清单追踪进度：
- [ ] 步骤 1：分析表单
- [ ] 步骤 2：创建字段映射
- [ ] 步骤 3：验证映射
- [ ] 步骤 4：填写表单
- [ ] 步骤 5：验证输出
```

### 条件工作流模式

引导通过决策点：

```markdown
## 文档修改工作流

1. 确定修改类型：
   **创建新内容？** → 遵循下方"创建工作流"
   **编辑现有内容？** → 遵循下方"编辑工作流"
```

### 反馈循环模式

对质量关键任务，实现验证循环：

```markdown
## 文档编辑流程

1. 进行修改
2. **立即验证**：`python scripts/validate.py output/`
3. 若验证失败：审查错误 → 修复 → 重新验证
4. **仅在验证通过后继续**
```

## 工具脚本

预制脚本相比生成代码有以下优势：
- 比生成代码更可靠
- 节省 token（上下文中无代码）
- 节省时间（无需生成代码）
- 确保一致性

```markdown
## 工具脚本

**analyze_form.py**：从 PDF 提取所有表单字段
python scripts/analyze_form.py input.pdf > fields.json

**validate.py**：检查错误
python scripts/validate.py fields.json
# 返回 "OK" 或列出冲突
```

明确说明 agent 应该**执行**脚本还是**读取**脚本作为参考。

---

## 需要避免的反模式

### 1. Windows 风格路径
- 使用：`scripts/helper.py`
- 避免：`scripts\helper.py`

### 2. 选项过多
- 差：令人困惑 —— "你可以用 pypdf，或 pdfplumber，或 PyMuPDF……"
- 好：提供默认值并留有余地 —— "使用 pdfplumber 提取文本。对于需要 OCR 的扫描 PDF，改用 pdf2image + pytesseract。"

### 3. 时效性信息
- 差："如果在 2025 年 8 月之前，请使用旧 API。"
- 好：使用"旧模式"章节区分当前方法和已弃用方法。

### 4. 术语不一致
选择一个术语并全程使用，不要混用（如 URL、路由、路径只选一个）。

### 5. 模糊的技能名称
- 好：`processing-pdfs`、`analyzing-spreadsheets`
- 避免：`helper`、`utils`、`tools`

## 技能创建工作流

### 阶段一：发现

收集以下信息：
1. 技能的目标和主要用例
2. 存储位置（个人 vs 项目）
3. 触发场景
4. 特定要求或约束
5. 现有示例或模式

### 阶段二：设计

1. 拟定技能名称（小写、连字符、最多 64 字符）
2. 编写具体的第三人称描述
3. 规划主要章节
4. 确定是否需要辅助文件或脚本

### 阶段三：实现

1. 创建目录结构
2. 编写带前置元数据的 SKILL.md
3. 创建辅助参考文件
4. 创建工具脚本（如需要）

### 阶段四：验证

- [ ] 描述具体并包含关键词
- [ ] 描述同时包含"做什么"和"何时用"
- [ ] 使用第三人称
- [ ] SKILL.md 正文在 500 行以内
- [ ] 全文术语一致
- [ ] 示例具体而非抽象
- [ ] 文件引用保持一级深度
- [ ] 无时效性信息
- [ ] 无 Windows 风格路径

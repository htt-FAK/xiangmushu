## Context

当前 `app.py` 已经把多个实验能力直接挂到主入口上：

- 模板视觉分析：`core/template_vision.py`
- 表格批量生成：`core/batch_generator.py`
- 内容审核与审核后重试：`core/content_auditor.py`
- 表格上下文与页图增强：`core/table_context.py`、`core/template_vision.py`
- 生成后视觉审核与自动优化：`core/visual_auditor.py`、`core/document_optimizer.py`

这会形成下面这种“重链路默认开启”的结构：

```text
模板上传
  -> 模板视觉分析
  -> 多模态/表格增强
  -> 任务分组
  -> 批量表格生成
  -> 内容审核
  -> 审核后重试
  -> 视觉审核
  -> 优化提示
```

而一期真正稳定的交付目标其实更接近：

```text
资料入库
  -> 锚点模板分析
  -> 分段检索生成
  -> Word 回填导出
  -> 人工复核
```

## Goals / Non-Goals

**Goals:**

1. 明确一期产品边界，避免主流程承载高风险实验能力
2. 收缩 `app.py` 主入口，只保留稳定演示所需的核心链路
3. 将复杂审核、视觉、多模型增强能力移出主流程，但暂不要求立即删除文件
4. 为后续实施提供模块级收缩清单与任务拆解

**Non-Goals:**

- 不在本提案中直接删除现有实验模块文件
- 不在本提案中重写 `core/generator.py` 或 `core/filler.py` 的内部算法
- 不否定后续二期继续研究视觉审核、表格增强或模型路由
- 不在本提案中改变知识库或 Word 填充的基础数据结构

## Decisions

### 决策 1: 一期主流程只保留四段式链路

**选择方案**：将产品主流程约束为四段式。

```text
┌────────────┐    ┌────────────┐    ┌────────────┐    ┌────────────┐
│ 知识库管理 │ -> │ 模板配置   │ -> │ 生成预览   │ -> │ Word 导出  │
└────────────┘    └────────────┘    └────────────┘    └────────────┘
      |                   |                 |                 |
      |                   |                 |                 |
  docx/pdf 入库       锚点扫描为主       分段 RAG         人工复核
```

解释：

- `tab_kb` 保留资料入库、来源管理、多知识库切换
- `tab_tpl` 保留 `.docx` 上传、锚点扫描、任务列表展示和缓存
- `tab_gen` 保留逐任务生成、结果预览、Word 回填、下载输出
- 首页与引导文案只描述上述主链路

### 决策 2: 模板分析的一期默认路径改为锚点优先且可强约束

**选择方案**：将带锚点模板作为一期主承诺路径。

```text
模板上传
  -> scan_anchor_tasks
  -> 识别到锚点: 直接生成 FillTask
  -> 未识别到锚点: 提示模板不在一期默认支持范围内
```

原因：

- 锚点模板把“找哪里该填”的问题变为确定性问题
- 可以显著降低无锚点复杂模板导致的误识别和定位错位
- 与一期“稳定导出初稿”的目标一致

保留空间：

- `TemplateAnalyzer` 中的无锚点分析代码可以继续存在
- 但不应再作为主流程默认路径或主卖点

### 决策 3: 将高风险增强能力移出主流程，而不是继续并列在侧栏开关中

**选择方案**：重分层，而不是一刀切删除。

#### 一期主配置

- `adv_use_stream`
- `adv_use_tavily`
- `adv_default_word_limit`
- `adv_top_k`
- `adv_retrieval_max_distance`
- `adv_force_clear_analysis`

#### 二期隐藏配置

- `adv_web_writing_mode`
- `adv_fast_gen`
- `adv_use_batch_table`
- `adv_use_mimo`

#### 从主流程移出的配置

- `adv_use_audit_agent`
- `adv_audit_regenerate`
- `adv_visual_audit_enabled`
- `adv_skip_template_vision`

### 决策 4: 以“保留文件、移出主入口”的方式处理实验模块

**选择方案**：模块不立即删除，但不再阻塞或污染一期主链路。

```text
一期保留模块:
  kb_registry / kb_extract / vector_store / template_analyzer / generator / filler

二期隐藏模块:
  content_auditor / batch_generator / task_grouper / table_context / vision_extract

移出一期路线模块:
  template_vision / visual_auditor / document_optimizer
```

原因：

- 可以保留研发成果，避免立刻产生删除争议
- 也能让回归测试和主流程收敛到真正可交付的范围

### 决策 5: 产品定位收缩为“初稿生成”，不承诺自动定稿

**选择方案**：所有首页文案、流程引导和后续规格均围绕“初稿”定位。

```text
定位:
  面向申报类 Word 模板的知识库驱动初稿生成工具

不再作为一期主承诺的内容:
  - 自动视觉审稿
  - 自动二轮修复
  - 复杂模板全自动识别
  - 复杂 Word 版式完全无损
```

## Risks / Trade-offs

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 一期范围收紧后，部分已有能力不再出现在主界面 | 用户误以为能力被删除 | 在设计文档和后续版本说明中明确“隐藏/实验”而非“移除文件” |
| 锚点模板要求提高了模板管理员配合成本 | 一期模板制作门槛上升 | 在首页和模板说明中明确推荐模板规范，减少无锚点误用 |
| 现有 `app.py` 强耦合多个增强模块 | 收缩实施时容易产生回归 | 先做 UI/流程分层，再处理内部依赖清理 |
| 二期隐藏模块长期无人维护 | 未来恢复成本上升 | 在任务中单独标注“保留但不进入回归主线”的模块清单 |

## Migration Plan

### 阶段 1: 明确一期边界与文案

1. 收缩首页、侧栏和标签页的产品叙事
2. 在 OpenSpec 中写明一期最小可交付范围
3. 明确哪些功能只是隐藏/实验，不属于主承诺

### 阶段 2: 收缩 `app.py` 主流程

1. 移除模板视觉分析作为默认分析路径
2. 移除审核、审核重试、视觉审核和自动优化的主流程入口
3. 将表格批量、页图和复杂多模型开关移到隐藏层或临时 feature flag

### 阶段 3: 收缩测试与验收口径

1. 让回归测试聚焦于入库、锚点模板、分段生成、Word 导出
2. 将视觉审核、MiMo、复杂模板等从一期验收门禁中移除
3. 更新文档和首页说明，统一“一期 = 初稿生成工具”的定位

## Open Questions

1. 一期是否允许保留“无锚点分析”作为明确标注的实验入口，还是应完全从 UI 中隐藏？
2. 一期上传器是否仍保留 `pptx/图片` 选项但不作为承诺，还是也应在界面层直接去掉？
3. 二期隐藏功能是保留在同一页面的开发开关里，还是迁移到单独的调试入口更合适？
4. 是否需要同步新增一个“模板规范说明”文档，帮助用户理解锚点模板的推荐写法？

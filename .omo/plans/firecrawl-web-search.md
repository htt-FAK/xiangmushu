**What you'll get:** 把项目里"联网搜索/联网补充"功能从"调用大模型（qwen3.7-plus + 百炼 enable_search）"整体换成 Firecrawl 的免费免 key 托管服务。生成时点"联网补充"开关，会直接调 Firecrawl 抓取网页证据塞进文档，不再产生任何大模型搜索的 token 费用。设置页里那个"联网搜索模型"下拉也会去掉——因为不再需要选模型了。

**Why this approach:** 你明确要"免 key"。Firecrawl 官方 Python SDK 必须带 API key，只有托管 MCP 端点（`https://mcp.firecrawl.dev/v2/mcp`）支持免 key 按 IP 限流调用。所以我们用项目已有的 `httpx`（保持现有版本锁定）自写一个极简的同步 MCP 客户端直连该端点，不引入新依赖、不改异步栈。失败时静默跳过联网、不影响主生成流程（与现有容错一致）。

**What it will NOT do:**
- 不升级 openai、不放宽 httpx 版本锁定、不引入异步框架或 MCP SDK 依赖。
- 不需要 Firecrawl API key（彻底零 key）。失败不回落到大模型搜索（否则费用又回来了）。
- 不删除 `VISION_WEB_MODEL`（视觉链路还要用）；不删除 `native_web_search` 数据字段（改 DB 风险大，留作恒 False）。
- 不动 leaderboard 基准工具和 smoke_test 诊断工具里的 enable_search 引用（它们是诊断用途，超出范围）。
- 不暴露成给外部 Claude/Cursor 调用的 MCP server（项目是确定性管线，没这需求）。

**Effort:** Medium（10 个 todo，5 个 wave）
**Risk:** Medium — 三个实现时实测/注意点：① 托管 Firecrawl MCP 是否要求会话握手（T1 先探测）；② 单 IP 共享限流（search 5次/分），多用户并发生成时可能被限流→T4 通过复用 SessionWebEvidenceCache 跨任务去重缓解，超限仍静默跳过；③ JSON-RPC error 响应（HTTP 200 + error body）需显式检查（T2 已指定）。
**Decisions to sanity-check:**
1. Firecrawl 接入用"自写极简 MCP client + httpx"而非官方 SDK（为了真正免 key + 不破坏依赖锁）。
2. 失败静默跳过、不回落 LLM 搜索（避免重新引入费用）。
3. `native_web_search` 字段保留但恒 False（最小改动，不碰 DB）。
4. `dashscope_chat.py` 只删 `for_enable_search` 死分支、不重构周边 fallback（控制爆炸面）。
5. 既有用户 DB 里 `model_choices.web_search` 孤儿行保留不迁移（读取时被忽略，无害）。
